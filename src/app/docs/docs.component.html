Use IAM Server as an OAuth2 and OpenID Connect provider.

IAM Server issues OAuth2/OIDC tokens through Spring Authorization Server and also exposes a separate JSON login API for first-party admin endpoints.

Overview

Two authentication surfaces

OAuth2 / OIDC browser flow

Use this for browser or SPA applications that need standard OAuth2 authorization code flow with PKCE and OpenID Connect identity tokens.

  • Authorization endpoint redirects unauthenticated users to /login.
  • Login establishes a Spring Security web session.
  • The saved authorization request resumes after successful login.

JSON JWT API

Use this for the server's own REST endpoints under /auth. It is not the same as the OIDC authorization-code flow.

  • POST /auth/login returns an API access token.
  • The refresh token is stored as an HTTP-only cookie scoped to /auth.
  • Protected API requests use Authorization: Bearer ....

Quickstart

Authorization Code + PKCE

This is the standard browser flow for admin-ui. The app generates a PKCE verifier, sends the user to the authorization endpoint, receives a code on the callback URL, then exchanges that code for tokens.

1. Create a PKCE verifier and challenge

const crypto = window.crypto;

function base64UrlEncode(buffer) {
  return btoa(String.fromCharCode(...new Uint8Array(buffer)))
    .replace(/\+/g, "-")
    .replace(/\//g, "_")
    .replace(/=+$/, "");
}

const verifierBytes = new Uint8Array(32);
crypto.getRandomValues(verifierBytes);

const codeVerifier = base64UrlEncode(verifierBytes);
const challengeBytes = await crypto.subtle.digest(
  "SHA-256",
  new TextEncoder().encode(codeVerifier)
);
const codeChallenge = base64UrlEncode(challengeBytes);

2. Redirect the browser to /oauth2/authorize

GET http://localhost:8080/oauth2/authorize
  ?response_type=code
  &client_id=admin-ui
  &redirect_uri=http%3A%2F%2Flocalhost%3A3000%2Foauth%2Fcallback
  &scope=openid%20profile%20email
  &state={random-csrf-state}
  &nonce={random-oidc-nonce}
  &code_challenge={codeChallenge}
  &code_challenge_method=S256
If the user does not have a Spring Security web session, IAM Server redirects to /login. After a successful login, Spring Security resumes the saved authorization request.

3. Handle the callback

http://localhost:3000/oauth/callback?code={authorization-code}&state={random-csrf-state}

Verify that the returned state equals the state value stored before redirecting. Then exchange the authorization code using the same code_verifier.

4. Exchange the code for tokens

curl -X POST http://localhost:8080/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "client_id=admin-ui" \
  -d "redirect_uri=http://localhost:3000/oauth/callback" \
  -d "code={authorization-code}" \
  -d "code_verifier={original-code-verifier}"

5. Expected token response

{
  "access_token": "eyJ...",
  "scope": "openid profile email",
  "id_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 300
}

Token Usage

Validate and use tokens

Discover server metadata

curl http://localhost:8080/.well-known/openid-configuration

Fetch signing keys

curl http://localhost:8080/oauth2/jwks

Call UserInfo

curl http://localhost:8080/userinfo \
  -H "Authorization: Bearer {oidc-access-token}"

ID token validation

  • Validate the RSA signature using /oauth2/jwks.
  • Validate iss against the configured issuer.
  • Validate expiration and issued-at times.
  • Validate nonce if one was sent in the authorize request.

Access token usage

  • Send as Authorization: Bearer {token}.
  • Use OIDC access tokens with OIDC resource endpoints such as /userinfo.
  • Use the JSON API token only with this server's protected REST endpoints.

JSON API

Server-owned JWT login API

These endpoints are separate from OAuth2/OIDC. They authenticate directly with email/password and issue this server's own HS256 API access token plus an HTTP-only refresh cookie.

POST /auth/loginAuthenticate with JSON email/password credentials.
POST /auth/refreshRotate the refresh token cookie and issue a new API access token.
GET /auth/meReturn the current API user from the bearer token.
POST /auth/logoutInvalidate the current refresh token cookie.

Login

curl -X POST http://localhost:8080/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "email": "user@example.com",
    "password": "user-password"
  }'

Login response

HTTP/1.1 200 OK
Set-Cookie: refresh_token=...; Path=/auth; HttpOnly; Secure; SameSite=Lax

{
  "accessToken": "eyJ...",
  "accessTokenExpiresAt": "2026-06-17T15:00:00Z"
}

Refresh

curl -X POST http://localhost:8080/auth/refresh \
  -b "refresh_token={refresh-token-cookie}"

Current user

curl http://localhost:8080/auth/me \
  -H "Authorization: Bearer {api-access-token}"

Logout

curl -X POST http://localhost:8080/auth/logout \
  -b "refresh_token={refresh-token-cookie}"

Errors

Common failure modes

invalid_client Wrong client_id, unsupported client authentication, or missing client registration.
invalid_grant Authorization code expired, already used, or does not match the PKCE verifier.
invalid_redirect_uri The redirect URI does not exactly match the registered URI.
403 on POST /login Missing or stale Spring Security CSRF token on the login form.
401 from /auth/* Invalid API credentials, invalid refresh token, or missing bearer token.

Local Development

Run locally

  1. Start Postgres with docker compose up -d postgres.
  2. Run the app with ./scripts/dev-run; it creates persistent development OAuth RSA keys under secrets/ when needed.
  3. Run the frontend with npm start, then open http://localhost:4200/docs for this page.
  4. Use an active seeded or created user account to complete the browser login step.
